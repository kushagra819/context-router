## 2. The Research Innovation

### What NOT to build
- A "query difficulty only" router (well-studied, not novel)
- A bloated multi-strategy framework
- A massive retrained DistilBERT classifier (too heavy for this scope)

### What TO build: **Confidence-Based Cascading + Stage-Aware Mixed-Tier Routing**

This is a two-layer innovation:

#### Layer 1: Confidence-Based Cascading (Primary)
- Start at Tier 1 (cheapest)
- After the verifier agent runs, extract a confidence signal from the verifier's output
- If confidence < threshold → escalate to next tier and re-run
- Continue until confident or Tier 4 (ceiling) is reached
- **Why this is novel for multi-agent systems**: Existing cascading routers work on single-model inference. Ours cascades across a full 3-agent pipeline (Analyzer→Solver→Verifier), using the verifier's judgment as the confidence signal.

#### Layer 2: Stage-Aware Mixed-Tier Routing (Secondary)
- Different agents use different tiers: e.g., Analyzer→Tier 2, Solver→Tier 3, Verifier→Tier 1
- The insight: not every agent stage needs the same capability. Decomposition is easier than solving; verification is easier than solving.
- This creates cost savings without quality loss.
- **Why this is novel**: Existing routers route entire queries. Ours routes within the agent pipeline stages.

#### Research Story
> Existing LLM routing approaches dispatch queries to models based on estimated difficulty. 
> Our system operates *inside* multi-agent workflows, using **workflow-stage-aware signals** — 
> specifically, verifier confidence and agent-role complexity — to dynamically select model tiers.
> This achieves a better quality-cost Pareto frontier than uniform-tier baselines, 
> while requiring no training data beyond the baselines we already collected.

### Why This Works for Your Project
1. **No additional model runs needed for the core router** — it can be evaluated by re-processing existing CSV data through routing logic
2. **Baselines become the comparison** — each single-tier baseline is a point on the Pareto frontier
3. **Ablation is clean** — turn off cascading, turn off mixed-tier, measure impact
4. **Presentation-ready** — Pareto plot of cost vs. quality with router vs. baselines is compelling
5. **Defensible** — the professor can ask "why not just use the best model?" and the cost analysis answers it

### 8.3 Central Logger Refactoring (Truncation Limit Removed)
* **File Modified:** [src/utils/logger.py](file:///c:/Users/Kumud/Desktop/Research/context-router/src/utils/logger.py)
  * **What:** Removed the 500-character truncation limit `[:500]` on `response_text` when logging LLM calls.
  * **Why:** The verifier agent's response details its step-by-step logic followed by a `Final Answer: [answer]` block at the very end. The 500-character cap truncated these responses, cutting off the critical `Final Answer:` pattern. This truncation physically removed the predicted answers from the CSV, rendering any offline F1 score recomputation or answer extraction impossible. Storing the full, untruncated response preserves the complete evaluation trail.

### 8.4 Standalone Recomputation and Correction Script
* **File Created:** [scripts/recompute_and_update_metrics.py](file:///c:/Users/Kumud/Desktop/Research/context-router/scripts/recompute_and_update_metrics.py)
  * **What:** Developed a script to validate baseline metrics directly from dataset ground-truth answers (GSM8K, HotpotQA, MuSiQue) and CSV columns.
  * **How it handles the data integrity issues:**
    1. **EM Accuracy:** Evaluated directly from the CSV's reliable `correct` column (computed in memory on full responses during the run), preventing any errors from truncated text.
    2. **F1 Calculation:** For correct responses, F1 is exactly `1.0`. For incorrect responses, it uses a regex to check for the required final answer patterns (`Final Answer:` or `answer is`). If present, it computes the true F1. If the pattern is missing (meaning it was cut off in the CSV), it is marked as truncated and defaults to a safe lower bound of `0.0`.
    3. **Strict Validation & Flagging:** Rather than using fragile length proxies or keeping plausible stored F1 numbers, the script enforces a fail-safe rule: if any baseline contains truncated verifier responses, it omits `"avg_f1"` (setting it to `null`) and writes `"needs_rerun": true` to the stats JSON. This strictly flags the configuration as corrupted and forces a clean baseline re-run.

### 8.5 Baseline Backup and Active Re-run Strategy
* **What:** Created a backup directory at [results/baselines_backup/](file:///c:/Users/Kumud/Desktop/Research/context-router/results/baselines_backup) and moved the 6 affected baseline CSV and JSON files (HotpotQA Tiers 2–4 and MuSiQue Tiers 2–4) there.
* **Why:** Since the response texts in the original CSV logs were permanently truncated to 500 characters, the true F1 scores for the resumed runs were lost.
